## Python Module Reload 陷阱 (2026-02-28)

**问题：** pytest fixture 使用 `importlib.reload()` 后，直接导入的对象不更新

```python
# 模块顶部
from emotiond.core import relationship_manager  # 绑定到旧对象

# fixture 中
importlib.reload(core)  # 创建新模块和新对象

# 测试中
relationship_manager.relationships["x"] = {...}  # 设置旧对象
generate_plan(request)  # 使用新对象 → 失败
```

**解决方案：**
- 使用模块引用 `core.relationship_manager` 而非直接导入
- Fixture 中 reload 整个模块链：config → db → core
- 确保所有引用指向同一 reloaded 模块

**适用场景：**
- pytest fixture 隔离测试数据库
- 动态配置切换（如 EMOTIOND_DB_PATH）
- 任何需要运行时重载模块的场景

## OpenEmotion MVP-3.1 Target-Specific Prediction (2026-02-28)

**项目路径：** `/home/moonlight/Project/Github/MyProject/Emotion/OpenEmotion`
**分支：** `feature-emotiond-mvp`
**Commit：** `36bdc57`

### 核心算法
```
pred_total = pred_global + α * pred_residual
α = n / (n + k)  # k=20, 收缩因子
```

### 新增功能
1. **target_id 解析**: `meta.target_id` → `meta.client_source` → `"default"`
2. **per-target residual learning**: 每个 target/action 独立学习偏差
3. **学习率拆分**: residual (0.1) vs global (0.02 = 0.1 * 0.2)
4. **解释扩展**: alpha, predicted_global/residual/total 字段

### 新增配置
| 参数 | 默认值 | 作用 |
|------|--------|------|
| `SHRINKAGE_K` | 20 | 收缩因子，n=k 时 α=0.5 |
| `LR_TARGET` | 0.1 | Residual 学习率 |
| `LR_GLOBAL_RATIO` | 0.2 | Global = residual × ratio |
| `EMA_DECAY` | 0.1 | 不确定性跟踪衰减 |

### 新增表
```sql
CREATE TABLE predicted_deltas_target (
    target_id TEXT NOT NULL,
    action TEXT NOT NULL,
    social_safety_delta REAL DEFAULT 0.0,
    energy_delta REAL DEFAULT 0.0,
    n INTEGER DEFAULT 0,
    ema_abs_error REAL DEFAULT 0.0,
    ema_sq_error REAL DEFAULT 0.0,
    UNIQUE(target_id, action)
);
```

## 串行子代理执行模式 (2026-03-03)

**问题**: 子代理回调可能丢失/延迟，导致串行流程中断

**解决方案**: SESSION-STATE.md 作为状态真相源 + 主动 poll

### 执行模式

```yaml
# SESSION-STATE.md
pending_workflow:
  active: true
  type: serial_subagents
  steps:
    - id: A
      runId: "xxx"
      status: running
    - id: B
      runId: null
      status: pending
  notify_on_done: "✅ 完成消息"
```

### 执行流程

1. `sessions_spawn` 子代理A
2. 更新 SESSION-STATE.md (A: running)
3. 等待回调 OR 主动 `subagents list` poll
4. A 完成 → 更新 (A: done) → spawn B
5. B 完成 → 更新 (B: done)
6. 发送 `notify_on_done`，设置 `active: false`

### 关键规则

- **不依赖回调**: 回调可能丢失/延迟/重叠
- **主动检查**: 用 `subagents list` 确认状态
- **SESSION-STATE.md 是真相**: 所有状态变更写入文件
- **防止重复通知**: 只有全部完成才发最终消息

### 回调处理规范 (AGENTS.md)

收到 `✅ Subagent ...` 回调时:
1. 读取 SESSION-STATE.md
2. 检查 `pending_workflow.active`
3. 如果 active: 更新步骤状态，判断是否继续
4. 如果全部完成: 发送 notify_on_done，设置 active: false

_HASH_EXCLUDE = frozenset({"generated_at", "report_path", "self_hash"})

score = (

| 规则状态 | active/deprecated/superseded | 比删除更安全 |
| Review Queue | 人工审核自动晋升 | 防偶然污染 |

### 有界评分模型

```python
HALF_LIFE_DAYS = 14

def recency_boost(days_ago):
    """
    有界 [0, 1]
    days=0 → 1.0
    days=14 → 0.368 (半衰)
    days=30 → 0.117
    """
    return exp(-days_ago / HALF_LIFE_DAYS)

def calculate_score(use_count, days_ago, importance, conflict_penalty):
    score = (
        2.0 * log(use_count + 1) +
        1.5 * recency_boost(days_ago) +   # 有界
        3.0 * min(max(importance, 0), 1) - # 有界
        2.0 * max(conflict_penalty, 0)
    )
```

### 规则状态机

```
active → deprecated (用户弃用)
active → superseded (被新规则替换)
deprecated → active (恢复)
```

### Review Queue 流程

```
3次独立纠正 → pending_review (待审核)
    ├─ 用户确认 → active RULE
    └─ 用户否决 → blocked (写 blocked 事件, 避免重复提议)
```
