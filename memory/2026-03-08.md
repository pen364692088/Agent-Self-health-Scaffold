## Key architectural finding

- In installed OpenClaw dist, `recordInboundSession()` is not the session selection point.
- The authoritative transport-level routing hooks are upstream at:
  - `resolveAgentRoute(input)`
  - `buildAgentSessionKey(params)`
- Therefore, session reuse/new-session behavior should be analyzed at route/sessionKey generation, not at post-routing recording.

## Probe result worth keeping

- Added `tools/session-route-probe` to separate:
  1. derived route/sessionKey behavior
  2. runtime session match behavior
- Current Telegram direct-message sample showed:
  - derived `session_key = agent:main:main`
  - runtime session match exists for `agent:main:main`
  - `reused_channel_slot = true`
  - `new_runtime_session_created = false`
- This is evidence that at least this sample is not a bottom-layer channel-slot churn case; apparent “new session” may be a higher-layer runtime/UI/session-surface distinction unless route inputs differ in another case.
