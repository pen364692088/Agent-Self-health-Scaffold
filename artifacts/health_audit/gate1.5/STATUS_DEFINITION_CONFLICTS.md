## 冲突 1: Shadow Mode 启用状态

### 定义漂移

| 来源 | 状态定义 |
|------|----------|
| Gate 1 报告 | "未启用" |
| 配置文件 | `enabled: true` |
| HEARTBEAT.md | "环境变量=true 或 配置文件 enabled=true" |
| 运行历史 | 有 SHADOW_TRACE.jsonl, SHADOW_SAMPLE_SUMMARY.json |

### 冲突分析

**Gate 1 判断依据**: 环境变量 `AUTO_COMPACTION_SHADOW_MODE` 未设置

**HEARTBEAT.md 定义**: 环境变量=true **或** 配置文件 enabled=true

**实际状态**: 配置文件 enabled=true，满足 HEARTBEAT 触发条件

### 结论

**状态定义漂移**

Gate 1 只检查环境变量，忽略了配置文件的启用状态。

正确判断: Shadow Mode **已启用**。

## 冲突 2: memory-lancedb 角色定位

### 定义漂移

| 来源 | 角色定义 |
|------|----------|
| Gate 1 报告 | "可忽略"，"冗余链或备用链" |
| 实际行为 | Gateway 每次对话都调用 |
| 调用频率 | 178 次/30min ≈ 6 次/min |

### 冲突分析

**Gate 1 判断**: 因为有 context-retrieve 替代，所以"可忽略"

**实际行为**: Gateway 内置记忆检索，每次对话都尝试调用

**正确角色**: **失效主链**

### 结论

**角色定义错误**

memory-lancedb 不是冗余链或备用链，是 Gateway 的主链记忆检索功能。

## 冲突 3: verify-and-close 流程状态

### 定义漂移

| 来源 | 状态定义 |
|------|----------|
| SOUL.md | "必须通过五工具链"，"强制" |
| TOOLS.md | "唯一收尾入口" |
| 实际执行 | 本轮任务未使用 |
| Gate 1 报告 | "待执行" |

### 冲突分析

**文档定义**: 必须强制执行

**实际状态**: 被绕过

**Gate 1 判断**: "待执行"（弱化处理）

### 结论

**状态定义弱化**

"待执行"掩盖了"主链被绕过"的严重性。

正确判断: **P1 问题** - 流程被绕过，缺少强制执行机制。

## 冲突 4: Execution Policy 健康状态

### 定义漂移

| 来源 | 状态定义 |
|------|----------|
| Gate 1 报告 | "已接线"，"健康" |
| MODULE_HEALTH_MATRIX | Wired: 50%, Effective: 75% |
| 样本状态 | 0 deny, 0 warn, 0 fallback |

### 冲突分析

**Gate 1 判断**: "已接线"→"健康"

**实际状态**: 工具链完整但样本为 0

**健康定义**: 缺少验证闭环是否算健康？

### 结论

**健康定义模糊**

"已接线"不等于"健康"。样本为 0 意味着无法验证实际效果。

正确判断: **已接线但缺少验证闭环**。

## 冲突总结

| 冲突项 | Gate 1 定义 | 正确定义 | 严重程度 |
|--------|-------------|----------|----------|
| Shadow Mode | 未启用 | 已启用 | 中 |
| memory-lancedb | 可忽略 | 失效主链 | 中 |
| verify-and-close | 待执行 | 主链被绕过 (P1) | 高 |
| Execution Policy | 健康 | 已接线但缺验证闭环 | 低 |

---

## 建议

1. 明确各模块的"启用"定义（环境变量 vs 配置文件）
2. 区分"主链"、"备用链"、"冗余链"
3. 区分"已接线"、"健康"、"有效"
4. 明确"待执行"与"被绕过"的区别
